01Executive Summary

AGENT_IITH uses a 4-agent adversarial debate architecture (Bull, Bear, Risk, Mediator) powered by Qwen2.5-72B-Instruct via Featherless.ai, with WebSocket streaming to a React frontend. Current research validates this approach — multi-agent debate improves factual accuracy and reasoning quality over single-model analysis, particularly for financial decisions requiring risk assessment. Key upgrade opportunities exist in debate turn structure, latency optimization, and real-time data pipeline hardening.

02Current Approach

Architecture:

  • Agent Structure: 3 adversarial agents (Bull/Bear/Risk) + 1 Mediator for synthesis
  • Model: Qwen2.5-72B-Instruct via Featherless.ai API
  • Data Flow: Yahoo Finance API + Bright Data scraper → Agent prompts → WebSocket → React UI
  • Storage: Supabase for conversation history and recommendations

Why This Stack:

  • Qwen2.5-72B offers strong reasoning at lower cost than GPT-4/Claude for hackathon constraints
  • Adversarial structure forces explicit trade-off analysis rather than generic summaries
  • WebSocket enables live streaming of agent reasoning chains to judges during demo

04Recommendations

1. Debate Turn Optimization (High Priority) Your current flow runs all 4 agents in parallel. Research suggests 2-turn debate improves accuracy:


Round 1: Bull + Bear generate initial positions


Round 2: Each agent critiques the other's reasoning


Round 3: Risk Agent flags unresolved conflicts


Round 4: Mediator synthesizes with explicit trade-off table

Source: DARE to Disagree framework shows multi-turn dialectic outperforms single-pass debate

2. Model Routing for Cost/Latency Keep Qwen2.5-72B for Mediator (needs deep reasoning), but consider smaller model for Bull/Bear agents:

  • Bull/Bear: Qwen2.5-7B or 14B (faster, cheaper, structured JSON output)
  • Risk: Qwen2.5-32B (balance of speed and caution)
  • Mediator: Qwen2.5-72B (final decision quality matters most)

Source: IEEE paper warns against "heavyweight backbones" for all agents in real-time systems

3. WebSocket Contract Hardening Add these fields to your streaming protocol:


{


  type: 'agent_update',


  agent: 'bull' | 'bear' | 'risk' | 'mediator',


  turn: number,


  confidence_delta: number,  // track confidence shifts during debate


  data_sources: string[],    // which scraped facts support this claim


  conflict_with: string[]    // which other agent's claims this challenges


}

Source: StockAPIs guide emphasizes "guaranteed delivery" and conflict tracking for audit trails

4. Rejected Approaches to Avoid

  • ❌ Don't switch to single-model analysis — research confirms debate structure adds value
  • ❌ Don't use REST polling for live data — WebSocket is correct choice
  • ❌ Don't add more than 4 agents — diminishing returns and latency issues
  • ❌ Don't use models smaller than 7B for financial reasoning — hallucination risk too high

05Next Steps

Before Forge Inspira Demo:

  1. Implement 2-turn debate structure — Modify agent prompts to accept previous agent outputs as input. Add turn counter to WebSocket events so UI can show debate progression.
  1. Add confidence tracking — Each agent should output confidence: 0-100 with each claim. Mediator should show how confidence shifted during debate (e.g., "Bull confidence dropped from 78→62 after Bear's revenue critique").
  1. Build fallback for API failures — If Featherless.ai times out, have cached analysis ready. Research shows judges penalize demos that fail on live data more than stale data.
  1. Prepare audit trail export — Export full conversation to PDF with timestamps, data sources, and confidence scores. This demonstrates transparency and helps judges follow reasoning chain.
